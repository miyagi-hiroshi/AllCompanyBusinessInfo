# ref が不安定（stale）になる原因と対策

## 現象

`browser_snapshot` で取得した ref を `browser_click` などに渡した際に、
「Element reference is stale (page has changed)」が発生し、操作が省略されていた。

## 原因の特定

### 1. 時間差（主因）

- **スナップショット**で ref を取得する時刻を T0、
- **クリック**を送信する時刻を T1 とする。
- T0 と T1 の間には、**ツール応答の往復・AI の処理・次のリクエスト送信**が含まれるため、数秒の遅延が発生する。
- この間にページの DOM が更新されると、T0 時点の ref が指していた DOM ノードは**既にアンマウント・差し替え**されている可能性がある。
- ブラウザ側は「この ref のノードはもう document に存在しない」と判断し、stale と返す。

### 2. React の再レンダー（具体的な要因）

対象画面（例: 受発注見込み入力）では次のような要因で再レンダーが起こり得る。

- **TanStack Query**: `useOrderForecasts` など複数クエリの完了・更新で `data` が変わり、コンポーネントが再レンダーされる。
- **useEffect による setState**:
  - `orderForecasts` / `filter` に依存する `useEffect` 内で `setLocalRows` が実行され、別フレームで再レンダーが発生する。
  - `shouldResetPage` の useEffect も setState の契機になり得る。
- **子コンポーネント**: `ExcelDataGrid` など大きな子が自身の state で再レンダーし、親まで含めて DOM が差し替わる可能性がある。

※ `queryClient` は `refetchOnWindowFocus: false`, `refetchInterval: false` のため、ポーリングやフォーカス時の再取得は原因ではない。

### 3. 結論

- **ref の「不安定さ」の正体**:  
  「スナップショット取得（T0）」と「操作実行（T1）」の\*\*時間差の間に React が再レンダーし、対象要素の DOM ノードが差し替わった」こと。
- 省略してよい理由ではなく、**操作手順とリトライで対処すべき事象**である。

## 対策（実行者側）

1. **操作の直前に必ずスナップショットを取る**
   - クリック・入力など ref を使う操作の**直前に** `browser_snapshot` を実行する。
   - その直後のスナップショットで得た ref だけを使って操作する（古い ref を使わない）。

2. **stale 時は 1 回リトライする**
   - 上記を守っても stale が出た場合は、**再度スナップショットを取得し、新しい ref で同じ操作を 1 回だけやり直す**。
   - 確認すべき項目を「ref が不安定だから」という理由で省略しない。

3. **操作と無関係なツール呼び出しを挟まない**
   - スナップショット取得とクリックの間に、スクリーンショット・長時間 wait・別画面遷移など、ページを変えうる処理を挟まない。
   - 「スナップショット → その ref でクリック」を一連の流れとして実行する。

## 追加で判明したこと（実測）

- **Toggle Sidebar（ref e2）** は、スナップショット取得後のクリックで **成功** する。
- **「2月」コンボボックス（ref e4）** は、**同じレスポンスで返ってきた直後の ref** を使っても、次のツール呼び出しでクリックすると **stale** になる。
- つまり「スナップショットとクリックの間の時間差」だけでなく、**メインコンテンツ内のコンボボックス（年度・月・プロジェクト）が属する DOM ツリーが、ツール 1 回の往復時間のあいだに再レンダーで差し替えられている**ことが原因。
- ヘッダー側のボタン（e2）は、グリッドや orderForecasts に依存しないため再レンダーが少なく、ref が有効なまま。

## ユーザーに助力をお願いしたい場合

**現状の MCP では「操作の直前にスナップショット＋リトライ」だけでは、フィルタのコンボボックスに対するクリックが安定しません。** ツール呼び出し 1 往復のあいだにメインコンテンツが再レンダーし、ref が無効になるためです。以下のいずれかの対応があると、フィルタ動作の自動確認が安定します。

1. **フィルタ部分の再レンダー抑制（推奨）**
   - `AdvancedFilterPanel` や受発注見込みページ内のフィルタ領域を、**グリッドの state（localRows 等）の影響を受けないように**する（例: フィルタ用 state を分離、またはフィルタを `React.memo` でラップし、props が同じなら再レンダーしないようにする）。
   - これにより、メインコンテンツの再レンダーが起きてもフィルタの DOM が差し替わりにくくなり、ref が有効な時間が延びる。

2. **data-testid の付与（将来用）**
   - フィルタの「年度」「月」「プロジェクト」に `data-testid`（例: `data-testid="filter-fiscal-year"`, `data-testid="filter-month"`, `data-testid="filter-project"`）を付与する。
   - 現行の MCP ブラウザは ref のみサポートのため即効はないが、**将来 MCP がセレクタ（例: `[data-testid="filter-month"]`）で要素を解決してクリックする機能をサポートした場合**、実行時ごとに要素を検索するため ref の有効期限に依存せず操作できる。

3. **MCP ブラウザ側の改善要望**
   - stale 時に「同じ role+name で再解決してリトライ」する、または「セレクタ（data-testid 等）でクリック」する API が使えると、確認作業が安定する。

## 参照
