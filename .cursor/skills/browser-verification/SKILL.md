# ブラウザ確認手順

## 🎯 目的

ユーザーにブラウザでの確認を依頼された時は、以下の手順で自律的にサーバー起動からブラウザ確認まで実行する。

## 📋 自律実行手順

### 1. サーバー状態確認と起動

```bash
# 既存のNode.jsプロセスを確認
tasklist | findstr node

# ポート5000の使用状況を確認
netstat -ano | findstr :5000
```

**サーバーが起動していない場合のみ**：

```bash
# 開発モードでサーバーを起動
npm run dev

# バックグラウンドで実行
# 起動状況を確認（8秒待機後）
Start-Sleep -Seconds 8; netstat -ano | findstr :5000 | findstr LISTENING
```

### 2. ログイン用テストアカウント

ログインが必要な確認では、以下のテスト用アカウントを使用する。**ローカル・開発環境の動作確認専用**であり、本番では使用しないこと。

| 項目           | 値                     |
| -------------- | ---------------------- |
| メールアドレス | `miyagi@cosmicb.co.jp` |
| パスワード     | `19brnZh77`            |

### 3. ログイン操作手順（固定・画面構成変更時以外は変更しない）

当システムのログインページ構成は固定のため、**キャプチャで入力方法を都度検討せず、以下をそのまま実行する**。

1. **遷移**: `browser_navigate` で `http://localhost:5000/` を開く。
2. **スナップショット1回**: `browser_snapshot` を1回だけ実行し、以下3要素の ref を取得する。
   - メールアドレス入力欄: `name: メールアドレス` の textbox の ref
   - パスワード入力欄: `name: パスワード` の textbox の ref
   - 送信ボタン: `name: ログイン` の button の ref
3. **入力（順序厳守）**:
   - 上記 ref を使い `browser_fill` でメールアドレス欄に `miyagi@cosmicb.co.jp` を入力する。
   - 続けて `browser_fill` でパスワード欄に `19brnZh77` を入力する。
4. **送信**: `browser_click` で「ログイン」ボタン（上記 ref）をクリックする。
5. **待機**: `browser_wait_for` で 2〜3 秒待つ。必要なら `browser_snapshot` でログイン後画面（URL または本文の変化）を確認する。

補足: スナップショットは「ref 取得のため1回」に留め、入力方法を考えるために複数回取得しないこと。入力方法は本手順で確定している。

### 4. ref が stale になった場合（確認の省略禁止）

- **「ref が不安定なため」という理由で、確認すべき項目を省略してはならない。**
- 原因を特定し、対策（操作直前にスナップショット・リトライ）を実施する。必要であればユーザーに助力を求める。
- 原因と対策の詳細は同フォルダの **REF-STALE-ANALYSIS.md** を参照する。

### 5. ブラウザでの確認

**サイドバー状態に注意**：

- サイドバーが表示されている（開いている）状態では、ヘッダーの Toggle Sidebar ボタンに `pointer-events: none` が付き**操作できない**。そのため「クリック成功」と返っても、実際にはボタンが操作できていない場合がある。
- **フィルタやメインコンテンツの操作確認は、サイドバーを閉じた状態で行う。** 必要なら `browser_press_key` で `Escape` を送り、サイドバーを閉じてから確認する。

**ブラウザツールが利用可能な場合**：

- 認証が必要な画面を確認する場合は、上記「3. ログイン操作手順」を実行してログインしてから、`browser_navigate` で対象URL（例: `http://localhost:5000/order-forecast`）へ遷移する。
- ログイン不要の確認（ログインページ表示のみ）の場合は `http://localhost:5000/` の表示確認のみ行う。

**ブラウザツールが利用できない場合**：

- ユーザーにブラウザツールの有効化を依頼する。
- 手動でブラウザを開き、上記テストアカウントでログインして確認を依頼する。

## ✅ 確認項目

### サーバー起動確認

- [ ] ポート5000でリスニング状態
- [ ] エラーログなし
- [ ] 開発モードで動作（ホットリロード有効）

### ブラウザ確認

- [ ] ページが正常に読み込まれる
- [ ] タイトルが「受発注見込み入力システム」と表示
- [ ] ログインページが表示される
- [ ] テスト用アカウントでログイン可能（必要時）
- [ ] エラーメッセージなし

## 🚨 トラブルシューティング

### ブラウザ確認中にエラーが発生した場合

1. **ターミナルでサーバーエラーを確認**
   - サーバーのログを確認
   - エラーメッセージがないかチェック
   - ポート5000で正常にリスニングしているか確認

2. **ターミナルを確認できない場合**
   - ユーザーにサーバーエラーの確認を依頼
   - ターミナルに表示されているエラーメッセージを報告してもらう

### ポート競合エラー

```bash
# すべてのNode.jsプロセスを停止
taskkill /F /IM node.exe

# 少し待ってから再起動
npm run dev
```

### 404エラー

- サーバーの起動順序を確認
- Viteの設定を確認
- 開発モードで正常に起動しているか確認

### ブラウザツールが利用できない場合

- **ユーザーにブラウザツールの有効化を依頼**
- 手動でブラウザを開いて確認を依頼
- 代替手段としてHTTPリクエストで確認：

```powershell
# PowerShellでHTTPリクエストを送信
$response = Invoke-WebRequest -Uri "http://localhost:5000/" -UseBasicParsing
$response.StatusCode
```

## 🔄 手順の特徴

- **自律実行**: ユーザーの指示なしに完全自動実行
- **ホットリロード対応**: 既存サーバーがあれば起動をスキップ
- **開発モード**: 開発効率を重視した設定
- **エラーハンドリング**: 各ステップでエラーを検出・対応
- **ユーザー協力**: ブラウザツール利用不可時の代替手段
- **確認項目**: 明確なチェックリストで品質保証
- **報告形式**: 統一された形式で結果を報告
