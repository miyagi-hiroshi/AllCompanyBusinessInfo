# 修正後のブラウザ確認手順

## 目的

修正実行後、ユーザーからブラウザでの動作確認を依頼されたときに、修正箇所の確認・デグレ有無の確認・表示データの確認・証跡キャプチャまでを自律的に実行する。

## 前提

- サーバーが未起動の場合は、先に `browser-verification` スキルの「サーバー状態確認と起動」に従って起動する。
- 修正対象の画面・URLが会話で特定できている場合はそのURLを、未特定の場合は修正したファイルからルートを推定して対象画面を特定する。

## ログイン用テストアカウント

ログインが必要な画面を確認するときは、以下のテスト用アカウントを使用する。**ローカル・開発環境の動作確認専用**であり、本番では使用しないこと。

| 項目           | 値                     |
| -------------- | ---------------------- |
| メールアドレス | `miyagi@cosmicb.co.jp` |
| パスワード     | `19brnZh77`            |

**注意**: 認証は既存システムDB（PostgreSQL）で行われるため、ログイン確認前に **PostgreSQL を起動しておく必要がある**。

## ログイン操作手順（固定・画面構成変更時以外は変更しない）

当システムのログインページ構成は固定のため、**キャプチャで入力方法を都度検討せず、以下をそのまま実行する**。

1. **遷移**: `browser_navigate` で `http://localhost:5000/` を開く。
2. **スナップショット1回**: `browser_snapshot` を1回だけ実行し、以下3要素の ref を取得する。
   - メールアドレス入力欄: `name: メールアドレス` の textbox の ref
   - パスワード入力欄: `name: パスワード` の textbox の ref
   - 送信ボタン: `name: ログイン` の button の ref
3. **入力（順序厳守）**:
   - 上記 ref を使い `browser_fill` でメールアドレス欄に `miyagi@cosmicb.co.jp` を入力する。
   - 続けて `browser_fill` でパスワード欄に `19brnZh77` を入力する。
4. **送信**: `browser_click` で「ログイン」ボタン（上記 ref）をクリックする。
5. **待機**: `browser_wait_for` で 2〜3 秒待つ。必要なら `browser_snapshot` でログイン後画面（URL または本文の変化）を確認する。

補足: スナップショットは「ref 取得のため1回」に留め、入力方法を考えるために複数回取得しないこと。入力方法は本手順で確定している。

## 実行手順

### 0. ログインから開始（認証が必要な場合）

認証が必要な画面を確認する場合は、**まずログイン画面でユーザーとパスワードを入力してログインすることから始める**。

- 上記「ログイン操作手順」に従ってログインを実行する。
- **ログイン成功** → 対象URLへ遷移し、以下 1 以降の手順を続ける。
- **認証エラー**（エラーメッセージ表示や 401/500 等）→ **MCPでアカウント・パスワードの妥当性を確認する**。手順は `browser-verification` スキルの「4. 認証エラー時のMCP確認」に従う（public スキーマの `users` / `employees` を SELECT で確認）。確認結果を報告してから必要に応じてユーザーに対応を依頼する。

### 1. 修正対象画面の特定

- ユーザーが「〇〇画面」「〇〇のページ」と指定している場合はその画面を対象とする。
- 指定がない場合は、修正したクライアントのルート（例: `client/src/routes/xxx.tsx`）やAPIのパスから、確認すべきURLを特定する（例: `/customers`, `/order-forecast`）。
- 対象URLをメモし、必要ならログインが必要なパスかどうかも確認する。

### 2. 表示データの準備（MCP利用）

- 対象画面が一覧・フォーム等で**データを表示する**場合、MCPでデータベースを確認し、表示されるべきデータが存在することを確認する。
- **Postgres MCP**（例: `mcp_postgres-neon-honban-public_list_tables`, `mcp_postgres-neon-honban-public_execute_query`）で、該当テーブルの存在や、表示想定のレコードが存在するかを確認する。必要なら検証用のSELECTを実行する。
- データが無い場合は、テスト用データを挿入するか、ユーザーに「表示確認用のデータがありません」と報告する。データを用意した場合はその旨を報告する。

### 3. ブラウザで対象画面を開く

- ブラウザツールで `http://localhost:5000/` に遷移する。
- ログインが必要な画面の場合は、上記「ログイン操作手順（固定）」を**そのまま**実行してログインし、続けて `browser_navigate` で対象URL（例: `http://localhost:5000/order-forecast`）に遷移する。
- ページ読み込み後、必要に応じて `browser_wait_for` で要素やテキストの出現を待つ。

### 4. 修正箇所の確認

- 修正した機能に該当するUI要素（ボタン・フォーム・テーブル・ラベル等）が表示されているか確認する。
- 修正内容（表示文言・表示/非表示・バリデーション等）が意図どおりか、スナップショットやスクリーンリーダー用の情報を参照して確認する。
- 問題があれば「修正箇所の確認結果」として箇条書きで報告する。

### 5. デグレード確認

- **同じ画面内**で、修正対象以外の既存要素が壊れていないか確認する（レイアウト崩れ・他ボタンの無効化・他項目の非表示など）。
- 必要ならスクロールして画面全体を確認する。
- デグレがあれば「デグレード確認結果」として箇条書きで報告する。問題なしなら「デグレは認められませんでした」と明記する。

### 6. データ表示の確認

- 一覧・テーブル・フォーム等でデータを表示している場合、手順2でMCPで確認したデータ（または既存データ）が画面に表示されているか確認する。
- 表示件数・主要項目の値が期待と一致するかを簡潔に確認する。
- 不一致やエラー表示があれば「データ表示確認結果」として報告する。

### 7. 証跡用キャプチャの取得

- 以下のキャプチャを取得し、ワークスペースに保存する（ツール例: `mcp_cursor-ide-browser_browser_take_screenshot`）。
  - **修正箇所が分かるキャプチャ**: 修正したUIが写っているもの（ファイル名例: `post-fix-{画面名}-修正箇所-{日時}.png`）。
  - **画面全体のキャプチャ**: デグレ確認用。必要に応じてフルページで取得する（`fullPage: true`）。
- 保存したファイルパスを「証跡キャプチャ」として報告に含める。

### 8. 報告

以下の形式で結果をまとめて報告する。

```markdown
## 修正後のブラウザ確認結果

### 対象画面

- URL: http://localhost:5000/xxxx
- 修正内容の要約: （一言）

### 修正箇所の確認

- （OK/NG と簡潔な所見）

### デグレード確認

- （問題なし / または発見した不具合の概要）

### データ表示確認

- （MCPで確認したデータが表示されているか、OK/NG と概要）

### 証跡キャプチャ

- 修正箇所: （保存したファイルパス）
- 画面全体: （保存したファイルパス、取得した場合）
```

## 注意事項

- MCPでDBを参照するときは、既存ルールに従い public スキーマは参照のみとし、データ変更は開発環境で必要な場合に限る。
- キャプチャには個人情報や機密データが含まれないよう注意する。含まれる場合は保存先を限定するか、マスキングを検討する。
- ブラウザツールが利用できない場合は、手順3以降をユーザーに依頼し、手順2（MCPでのデータ確認）と報告形式だけ実行する。
